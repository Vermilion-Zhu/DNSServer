# dnslib
## DNSServer
DNSServer：__负责网络通信的开启、维护、关闭。是整个服务的入口。__ socketserver 的封装类（在大多数情况下，你只需要向其传递一个适当的解析器实例 resolver，并在前台或后台启动即可）。

DNSHandler：__负责原始数据的解析和封包，会从套接字中读取原始二进制比特流，解码并把请求转交给 Resolver 处理，收到 Resolver 给出的结果后，把二进制格式发给用户。__ 由 DNSServer 实例化用于处理请求的处理器。其 handle 方法负责发送和接收数据包（处理 TCP 长度前缀），并将协议处理委派给 get_reply。该方法会解码数据包，将 DNSRecord 传递给解析器实例，并对返回的 DNSRecord 进行编码。

注：除非你需要在解析器中获取原始协议数据，否则通常不需要修改 DNSHandler。

DNSLogger：__可以通过自定义 logf 函数决定日志是打印还是写入文件。__ 该类为 DNSServer 实例处理请求的各个阶段提供了一组默认的日志函数。这些功能可以通过 log 类变量中的标志位来启用或禁用。

Resolver (解析器)：__接收一个 DNSRecord 请求，根据自己实现的逻辑（查询数据库、查文件或直接返回固定的 IP 构造并返回另一个 DNSRecord 响应）。__ 实现 resolve 方法的实例。该方法接收解码后的请求数据包并返回响应。

注：要实现自定义解析器，通常只需实现此接口即可。另外，由于解析器只有一个单例实例，因此需要注意线程安全和阻塞问题。

## 必做
1. 要求 UDP 53 端口
2. 目前实现了递归查询，但是没有实现迭代查询。且有一些问题：
   1. 目前没有 TTL 过期时间，缓存永远不会失效。
   2. add_records() 多数把 rr.rdata 转为字符串，导致 MX 这样的记录结构会被破坏
   3. _build_reply() 对 MX/TXT/复杂类型 可能出错
   4. key 与 rname 可能不一致：把缓存以查询的 qname 为键，有时应以 rr.rname 为键保存真实记录名
   5. 线程安全：newrecord、计数器等无锁保护，DNSServer 多线程时会有竞态。
   6. 只用 UDP、固定 512 字节接收：可能错过被截断或需 TCP 的响应。
   7. 无负缓存、无容量控制、无持久化。
3. dnslib.client 应该已经满足自定义指定 DNS 服务器和现实详细内容（3个）的要求，我们可能只需要包装一下。
